## 2. 项目一句话定位

DeepCast 是一个“输入主题，自动完成深度研究并生成双人对谈播客”的 AI Agent 产品 Demo。它面向 AI 产品经理作品集展示，重点不是证明某个单点技术，而是展示从用户痛点、竞品分析、Agent 工作流、内容质量控制、TTS 体验到产品化指标设计的一整套产品思考。

当前主线：

```text
主题输入
  -> 任务规划
  -> Tavily + SerpApi 混合搜索
  -> 搜索结果过滤与权威性排序
  -> 并行研究摘要
  -> 迭代式深度搜索与信息饱和判断
  -> 报告大纲 + Self-Refine
  -> 节目蓝图
  -> 双人播客脚本
  -> MiMo TTS 导演模式语音合成
  -> FFmpeg 拼接成 MP3
```

核心用户价值：把需要长时间阅读的复杂资料，转换成可以在通勤、运动、家务场景中收听的知识播客，同时保留 Markdown 研究报告供深读。

## 4. 当前重要代码结构

### 4.1 后端

主要目录：

```text
backend/src/
  agent.py               # DeepResearchAgent 总编排器
  agents/                # 多 Agent 抽象与执行入口
  services/              # 底层能力库
  config.py              # 环境变量与运行配置
  main.py                # FastAPI + SSE 接口
  models.py              # 核心数据模型
  prompts.py             # 中文 Prompt 模板
  utils.py               # 来源格式化等工具函数
```

`agents/` 的定位：

- `DirectorAgent`：注册和分派专业 Agent。
- `PlannerAgent`：任务拆解、搜索补充任务、信息增益判断。
- `ResearcherAgent`：搜索、过滤、权威性排序、摘要。
- `WriterAgent`：报告、节目蓝图、播客脚本。
- `CriticAgent`：报告质量评审。
- `BaseAgent` / `AgentResult`：统一接口。

`services/` 仍然存在是刻意保留的分层：

- `agents/` 是“业务角色层”，用于体现多 Agent 产品/架构设计。
- `services/` 是“能力实现层”，负责 LLM 调用、搜索、报告、脚本、音频等具体能力。
- 不建议为了形式上“只剩 agents”而删除 `services/`。除非后续明确要大重构，否则维持当前分层更稳。

### 4.2 前端

主要目录：

```text
frontend/src/
  App.vue                     # 全局状态与 SSE 事件路由
  services/api.ts             # fetch + ReadableStream 的 SSE 客户端
  components/SetupView.vue    # 主题输入
  components/ProductionView.vue # 制作流程、节目蓝图、错误终态
  components/PlayerView.vue   # MP3 播放器与报告阅读
  components/TerminalLog.vue  # 实时日志
```

前端重点体验：

- 制作过程透明化：展示搜索、任务、来源、报告精炼、节目蓝图、音频进度。
- 播放器产品化：把试听做成更接近 MP3/播客播放器的形态。
- 错误终态：网络波动或后端异常后，不再无限停住，而是进入 `error` 状态。
- SSE 事件 union type：`ResearchStreamEvent` 已从宽泛对象收敛为联合类型，便于前端维护。

## 7. 历史对话主线

下面按时间线概括我们围绕这个项目做过的关键工作。

### 7.1 初始化与项目理解

- 创建并维护了 `CLAUDE.md` / `AGENTS.md` 类项目协作说明。
- 梳理项目目标：自动播客生成 Agent，不只是技术 Demo，而是面向 AI 产品经理作品集的产品项目。
- 明确项目要从“技术项目”包装为“产品项目”：用户痛点、竞品、体验指标、商业化路径、产品迭代优先级都要能讲。

### 7.2 TTS 从旧模型迁移到 MiMo

曾经的旧 TTS 效果不理想，后来切到 MiMo-V2.5-TTS：

- 默认 TTS 服务改为小米 MiMo 接口。
- TTS 调用方式从普通 HTTP/MP3 改成 OpenAI 兼容 chat completions，返回 base64 WAV/音频数据。
- 音频合成逻辑适配 WAV 输入，最终仍可用 FFmpeg 输出 MP3。
- 增加 `verify_mimo_tts.py` 验证脚本。

后续进一步优化了拟人化语音：

- 引入 MiMo 导演模式：按“角色 / 场景 / 指导”生成语音风格指令。
- 引入 VoiceDesign：用自然语言描述音色，而不是只用预置声音。
- 引入 audio tags：如 `(轻笑)`、`(叹气)`、`(语速加快)`，增强局部语气。
- 播客角色从旧的 `liwa/xiayu` 改为中文角色，并最终稳定为 Host `苏打`、Guest `茉莉`。
- Guest 曾出现语速过慢，原因判断为“冰糖”音色和嘉宾定位不匹配；后续改为更知性、节奏更合适的“茉莉”，并在 Prompt 和导演指令里要求语速适中偏快、节奏明快。

### 7.3 产品化文档与作品集叙事

围绕 `docs/JD.md` 中的阿里、字节等 AI 产品经理岗位要求，曾经补过一批产品化材料：

- PRD / 用户旅程 / 竞品分析 / 产品优化路线图 / 前后端设计文档。
- 强调 AI PM 能力：用户研究、竞品分析、指标设计、Agent 流程设计、Prompt Engineering、模型能力边界、商业化思考。
- 后来为降低 `docs/` 噪音，把阶段性文档精简，只保留核心文档和对外更有用的竞品分析。

当前作品集叙事应集中在：

- 用户问题：碎片化时间无法消费深度知识。
- 产品方案：主题到双人播客 + Markdown 报告。
- 差异化：相比 NotebookLM 更强调中文、实时搜索、可控对话风格；相比传统播客平台更强调按需生成。
- 产品指标：WAST、TTFA、完播率、生成失败率、等待流失率。
- 技术服务于体验：Terminal UI、节目蓝图、Self-Refine、搜索透明化、TTS 导演模式。

### 7.4 HelloAgents 替换与多 Agent 包装

项目早期依赖 HelloAgents 框架，后来为了简历和技术可控性，移除了 HelloAgents：

- 用 OpenAI SDK 直接调用 LLM。
- 新增 `services/llm.py` 统一封装普通调用、流式调用、JSON 输出。
- 新增 `services/note_manager.py` 替代 HelloAgents NoteTool。
- 搜索层改为直接调用 Tavily / SerpApi，而不是 HelloAgents SearchTool。
- planner / summarizer / reporter 改为直接使用 OpenAI SDK。

之后又补回“多 Agent”的产品包装，但不是回到 HelloAgents：

- 新增 `agents/` 作为真实执行入口。
- `DeepResearchAgent` 负责阶段编排、并发和 SSE。
- `DirectorAgent` 负责分派 Planner / Researcher / Writer / Critic。
- `services/` 保留为底层实现库。

这是当前最重要的架构判断之一：保留多 Agent 叙事，但不要为了框架名牺牲可控性。

### 7.5 DeepSeek API 切换

用户后来手动把主模型切换到 DeepSeek API，并完成全链路测试。围绕 DeepSeek 做过这些调整或 review 关注点：

- 模型配置收敛：不再写多套 `SMART_LLM_MODEL` / `FAST_LLM_MODEL`。
- 结构化输出使用 DeepSeek JSON Output + 本地 schema validation，而不是继续宣称 XGrammar token 级约束。
- DeepSeek thinking mode / JSON mode / multi-round / tool calls 文档曾作为 code review 依据。
- 长耗时任务要配置更长 timeout，并对网络波动做重试。

当前不要再把 ECNU 作为主模型叙事。ECNU 只可能作为历史残留或旧文档背景。

### 7.6 深度搜索与报告质量优化

参考 `deep-research`、`research-writer`、`podcast-blueprint` 等 skill 的思想，但不是直接使用这些 skill，项目中做过这些优化：

- 增加“节目蓝图”阶段：在正式生成脚本前，先约束 Hook、分段、转场、问题链和 CTA。
- 报告撰写前生成结构化大纲。
- 报告生成后加入 Critic 的 Self-Refine：初稿 -> 批判 -> 修改。
- 深度搜索加入迭代反思：判断已有覆盖、找信息缺口、补充搜索任务。
- 保持原有 Tavily + SerpApi 混合搜索模式，不切到单一搜索或纯 RAG。
- 搜索结果加入 LLM 质量过滤和域名权威性排序。
- 增加信息增益/重复度判断，避免无限搜索。

### 7.7 前端体验优化

前端围绕“等待可感知”和“播客产品感”做过多轮优化：

- TerminalLog 展示实时 Agent 动作。
- ProductionView 展示制作流程。
- 节目蓝图在前端可见，提升用户对内容结构的预期。
- 播放试听界面改成更接近 MP3/播客播放器，并放到制作流程面板下方/节目蓝图面板左侧的布局方向。
- 网络波动时不再一直停住：增加前端连接重试、idle timeout、`error` 终态。
- SSE 事件定义成 TypeScript union type，减少后续维护风险。

### 7.8 网络波动、取消和超时

历史上出现过 DeepSeek API `APIConnectionError: Connection error`，导致前后端一直停着。后续优化方向：

- OpenAI SDK 客户端自身 `max_retries=0`，项目层做可控重试。
- LLM、TTS、搜索分别有最大重试次数配置。
- 长耗时 LLM 任务使用 `LLM_LONG_TIMEOUT`。
- SSE 有 heartbeat。
- 前端读取流有 idle timeout。
- `cancelProduction()` 先通知后端 `/research/cancel`，再 abort 前端连接。
- 后端工作流有 `WORKFLOW_MAX_SECONDS` 总时长保护。

迁移后如果再遇到卡住，优先检查：

1. DeepSeek API 是否网络不稳。
2. SSE 是否仍在收到 heartbeat。
3. 前端是否进入 `error` 终态。
4. 后端 cancel event 是否被长耗时阻塞调用及时感知。

## 8. 关键技术决策记录

### 8.1 为什么不换成 LangChain / CrewAI

曾经讨论过是否把 HelloAgents 换成业界更常用框架。结论是：

- 不建议为了简历写“用了某框架”而引入复杂框架。
- 当前工作流是清晰的 Plan-and-Solve，更适合轻量自研编排。
- 直接 OpenAI SDK + 自研 Agent 抽象更可控，也更容易解释产品逻辑。

对外可以说：

> 没有选择重框架，而是基于 OpenAI SDK 自研轻量多 Agent 编排，保留 Planner / Researcher / Writer / Critic 的职责边界，避免框架抽象掩盖产品流程。

### 8.2 为什么保留 `services/`

`services/` 不是冗余，而是能力实现层。`agents/` 调用它们，提供产品化的角色抽象。

推荐对外表达：

> 项目采用“Agent 角色层 + Service 能力层”的分层。Agent 层承载任务分工和产品叙事，Service 层沉淀可复用能力，避免把搜索、报告、TTS 等实现细节塞进单个 Agent 类。

### 8.3 为什么不做 Native RAG

项目主输入是任意主题，不是固定私有知识库。当前更适合 Deep Search：

- Tavily + SerpApi 保证时效性和广覆盖。
- LLM 过滤和权威性排序提高来源质量。
- 迭代式搜索弥补单轮搜索覆盖不足。
- MemoryManager 只作为可选扩展，不把项目过早改成向量库/RAG 产品。

### 8.4 为什么节目蓝图很重要

直接“报告 -> 对话脚本”容易生成机械问答。节目蓝图先决定：

- 开场 Hook。
- 内容分段。
- 听众问题链。
- 转场方式。
- 情绪节奏。
- 收尾 CTA。

这能把播客脚本从“报告改写”推进到“节目策划”。

### 14.3 节目蓝图和精炼任务 schema 兜底

用户反馈后端日志中出现：

```text
Structured output failed schema validation: $.sections: expected at most 3 items, got 4
Agent 'writer' failed: {'blueprint': None}
```

根因是模型生成了 4 个节目段落，但 schema 要求最多 3 个，导致整个蓝图被丢弃。类似地，搜索精炼有时会生成 9 个补充任务，但 schema 最多允许 3 个。

后续修复：

- `call_llm_json()` 增加 `response_transform` 钩子，在 JSON 解析后、schema 校验前允许业务层做规范化。
- `ScriptGenerationService._normalize_blueprint()`：节目蓝图超过 3 段时截断到前 3 段。
- `PlanningService._normalize_refine_result()`：补充搜索任务超过 3 个时截断到前 3 个。
- 保留 schema 对字段类型、必填字段和枚举的校验，不全局放松结构化输出。

这是一个很重要的工程原则：模型输出“多给几项”属于可规范化错误，不应该直接让用户界面缺一整块。

### 14.4 搜索精炼轮次语义修复

用户发现每次都会出现“已达到最大搜索轮次（2轮），虽仍存在多个知识缺口，但根据限制无法继续搜索”。这不是简单搜索不够，而是 prompt 中“当前搜索轮次：2 / 最大轮次：2”让模型误解为“本轮不能再搜”。

修复点：

- `research_analyzer_instructions` 把“当前搜索轮次”改为“当前分析轮次”。
- Prompt 新增：
  - 本轮是否仍可发起补充搜索。
  - 本轮完成后剩余补充搜索轮次。
  - 明确“不要因为当前分析轮次等于最大轮次就提前放弃本轮搜索”。
- `agent.py` 对最后一轮执行完后的状态做显式区分：
  - 如果信息重复度过高，仍然智能终止。
  - 如果最后一轮已执行完但仍可能有缺口，发出“搜索预算已用完，进入报告阶段”，不再误写成“信息饱和”。
- 日志语义从“信息增益”统一为“信息重复度”，避免理解反了。

迁移后如果要继续调搜索，不建议先把 `MAX_RESEARCH_REFINE_ROUNDS` 盲目调大。先看第 2 轮是否真的发起补充任务，再决定是否调成 3。

### 14.5 报告失败兜底

用户提供了一份异常 note：`backend/output/notes/note_20260526_204151_1.md`。报告标题是“纯血鸿蒙的发展史”，但正文变成“关于 AI 报告生成失败的分析与改进报告”，导致下游播客也开始分析“AI 报告生成失败”。

根因：

- `ReportingService.generate_report()` 在模型返回空文本时曾返回自然语言占位文案 `报告生成失败，请检查输入。`
- 后续 Critic / Writer 把这个占位文案当真实报告继续精炼。
- 模型于是认真写了一篇“AI 报告生成失败”的改进报告，并继续进入播客脚本链路。

修复点：

- `reporter.py` 新增 `REPORT_GENERATION_FAILURE_MESSAGE` 和 `is_report_generation_failure()`。
- 报告生成空文本时不再返回自然语言失败占位，而是返回空字符串。
- 报告初稿为空或失败占位时，跳过 Critic / Writer refine。
- `DeepResearchAgent._generate_report_via_agents*()` 拿到空报告或失败占位时直接抛错，中断后续 note 持久化、节目蓝图、脚本和音频。
- `WriterAgent` 也会把失败报告标记为 `success=False`。

迁移后如果再出现类似“报告生成失败主题被下游加工”的问题，优先检查 `is_report_generation_failure()` 是否覆盖了新的异常文案。

### 14.6 搜索 API timeout 解释

用户曾问：

```text
Tavily search failed (TimeoutError), retrying 1/2 in 1.0s
SerpApi search failed (ConnectTimeout), retrying 1/2 in 1.0s
```

结论：

- 这是搜索 API 的临时网络超时，不一定代表配置或代码坏。
- 当前 `SEARCH_API=hybrid` 时，每个查询会打 Tavily / SerpApi 并合并结果。
- 只要后续出现 `Search backend=hybrid results=...` 且结果数大于 0，说明最终搜索成功。
- 如果频繁超时，可以考虑给搜索请求单独设置更长 timeout，或临时切到 `tavily` / `serpapi` 单后端排查。

这个解释适合写进排障说明，但不必放进 README 主体。
