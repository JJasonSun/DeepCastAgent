# DeepCast 产品优化路线图

> 面向 AI 产品经理视角的深度优化规划，聚焦可落地的技术-产品协同迭代方向。

---

## 一、信息收集：从"单轮搜索"到"深度研究"

### 现状

当前 DeepCast 的搜索流程是 **单轮规划 → 并行执行**：Planner 一次性生成 3-5 个搜索任务，每个任务独立执行一次混合搜索（Tavily + SerpApi），汇总后直接进入报告生成。这意味着：

- 搜索查询在执行前就已固定，无法根据中间结果动态调整
- 缺少信息质量评估，搜索结果直接进入上下文
- 无法识别"信息饱和"状态，可能遗漏关键维度

### 优化方向 1.1：迭代式查询精炼 ✅ 已实现

**核心思路：** 让 Agent 像研究员一样，先粗后细、逐步收敛。

```
第 1 轮：宽泛探索（"AI 播客市场现状"）
    ↓ 分析返回结果，发现关键实体
第 2 轮：定向深挖（"NotebookLM 用户增长数据 2025"）
    ↓ 发现信息缺口
第 3 轮：补充验证（"AI 播客生成 技术壁垒 专利"）
    ↓ 信息饱和 → 终止
```

**产品价值：** 搜索深度从 1 轮提升到 3-5 轮，信息覆盖率显著提高。对用户而言，报告的专业度和可信度是核心体验指标。

**技术要点：**
- 每轮搜索后，LLM 进行递归反思：总结覆盖情况、识别知识缺口、规划下一轮搜索策略
- 动态生成新的搜索关键词，避免与历史查询重复
- 明确下一轮优先来源类型（官方文档、论文、行业报告、头部媒体等）
- 终止条件：信息饱和（连续 N 轮无新增有价值信息）或达到轮次上限

**实现要点：**
- `planner.py` 新增 `analyze_and_refine()` 方法，使用 `REFINE_JSON_SCHEMA` 结构化输出
- `REFINE_JSON_SCHEMA` 包含 `coverage_summary`、`knowledge_gaps`、`next_search_strategy`、`source_strategy`
- `agent.py` 新增 `_stream_refine_phase()` 和 `_refine_research()` 迭代精炼循环
- 配置项：`max_research_refine_rounds`（默认 2）
- SSE 事件：`refine_round`、`refine_saturation`

### 优化方向 1.2：搜索结果质量评估 ✅ 已实现

**核心思路：** 不是所有搜索结果都值得进入上下文，需要一个"筛选门"。

**产品价值：** 过滤噪音信息，减少 LLM 上下文中的无效 token，提升报告质量的同时降低推理成本。

**技术要点：**
- 对每个搜索结果（标题 + 摘要 + URL），LLM 判断其与研究主题的相关性
- 评估维度：内容相关性、信息新颖性、来源权威性
- 已访问 URL 去重，避免重复劳动
- 被判定为"无用"的结果不进入上下文

**实现要点：**
- `services/search.py` 新增 `filter_search_results()` 函数，使用 Fast 模型 + `FILTER_JSON_SCHEMA`
- 评估维度：相关性、信息密度、来源质量
- 配置项：`enable_search_filter`（默认 True）
- 在 `agent.py` 的 `_execute_task()` 中，搜索后、摘要前调用过滤

### 优化方向 1.3：信息源可信度分层 ✅ 已实现

**核心思路：** 不同来源的信息权重不同，权威来源应优先采纳。

**产品价值：** 提升报告的可信度和专业性，减少"幻觉"风险。

**技术要点：**
- 维护信息源白名单（学术期刊、官方统计局、头部媒体等）
- 对搜索结果的域名进行权威性评分
- 在上下文拼接时，高权威来源排在前面，低权威来源标记"待验证"

**实现要点：**
- `services/search.py` 新增 `DOMAIN_AUTHORITY` 域名权威性白名单（学术 10 分、头部媒体 8 分、技术社区 6 分、一般 3 分）
- `score_source_authority(url)` 根据域名返回权威性评分
- `sort_by_authority(results)` 按权威性降序排列搜索结果
- 在 `agent.py` 的 `_execute_task()` 中，搜索过滤后、上下文构建前调用排序
- 零 LLM 成本，基于规则评分

## 二、内容生成：从"一次成型"到"迭代精炼"

### 现状

当前报告生成是一次性调用 Smart LLM，将所有研究摘要拼接后直接输出 Markdown。脚本生成也是一次性调用 Fast LLM，输出对话 JSON。这种方式依赖单次推理的质量，缺乏自我纠错能力。

### 优化方向 2.1：报告迭代精炼（Self-Refine） ✅ 已实现

**核心思路：** 生成 → 批判 → 修改，循环直到质量达标。

```
草稿生成（Author Agent）
    ↓
自我批判（Critic Agent）→ 评估维度：逻辑严谨性、数据支撑度、语言专业性、是否存在矛盾
    ↓ 反馈意见
迭代修改（Author Agent）→ 基于批判意见修改草稿
    ↓ 重复 2-3 轮
最终报告
```

**产品价值：** 报告质量从"能用"提升到"专业"，这是竞品分析中拉开差距的关键点。面试时可以展示"AI 质量自闭环"的产品设计思路。

**技术要点：**
- 正式写作前先生成报告大纲，明确读者问题、核心主线、章节论点和证据需求
- Critic Agent 使用独立的 system prompt，角色设定为"严格的行业分析师"
- 评估结果为结构化反馈（问题列表 + 修改建议）
- 迭代轮次可配置（默认 2 轮），避免过度消耗 token

**实现要点：**
- `services/reporter.py` 新增 `REPORT_OUTLINE_JSON_SCHEMA`，报告生成前先形成结构化大纲
- `services/reporter.py` 新增 `generate_report_with_refine()` 和 `generate_report_with_refine_stream()`
- `_critique_report()` 使用 Smart 模型 + `CRITIC_JSON_SCHEMA` 进行结构化评估
- `_refine_report()` 根据 Critic 反馈修改报告
- `prompts.py` 强化报告写作规则：TL;DR、关键发现表、证据链、来源风险、待验证问题
- 配置项：`enable_report_outline`（默认 True）
- 配置项：`max_report_refine_rounds`（默认 1）
- SSE 事件：`report_refine`（含评分和 verdict）

### 优化方向 2.2：脚本情感丰富度提升 ✅ 已实现

**核心思路：** 让对话脚本从"信息传递"升级为"情感表达"。

**产品价值：** 播客的核心体验是"听感"。情感丰富的脚本直接决定了用户是否愿意听完。

**技术要点：**
- 先生成节目蓝图，明确 Hook、主体分段、Host 追问、转场逻辑和 CTA
- 脚本 JSON Schema 扩展 `emotion`（情绪状态）和 `audio_tag`（音频标签）字段
- TTS 使用导演模式 style 指令，每句台词根据 emotion 动态生成风格描述
- 音频标签（轻笑、叹气、语速加快）嵌入文本，实现词级控制

**实现要点：**
- `script_generator.py` 新增 `SCRIPT_BLUEPRINT_JSON_SCHEMA`，先生成结构化节目蓝图，再基于蓝图生成对话
- `script_generator.py` 的 `SCRIPT_JSON_SCHEMA` 包含 `emotion` 和 `audio_tag` 字段
- `prompts.py` 的 `script_writer_instructions` 指导 LLM 生成丰富的 emotion（含语速节奏）和 audio_tag
- `audio_generator.py` 使用导演模式三段式指令（角色/场景/指导），根据 emotion 动态构建
- 音频标签通过 `_embed_audio_tag()` 嵌入文本开头
- VoiceDesign 模式使用文本描述生成音色，支持自然语言控制语速和风格

## 三、系统架构：从"单 Agent"到"多智能体协作"

### 现状

当前是单个 `DeepResearchAgent` 编排所有步骤，虽然内部调用了多个服务，但本质上是一个线性流程的协调器，缺少专业化的分工和记忆管理。

### 优化方向 3.1：多智能体分工 ✅ 已实现

**核心思路：** 将现有服务层抽象为正式的 Agent 接口，引入 Director 协调器。

| 角色 | 职责 | 模型层级 | 实现 |
|------|------|----------|------|
| PlannerAgent | 任务分解、信息增益分析 | Smart | 包装 PlanningService |
| ResearcherAgent | 执行搜索、信息筛选、权威性排序、摘要 | Fast | 包装 search + summarizer |
| CriticAgent | 评估报告质量、提出修改意见 | Smart | 包装 ReportingService |
| WriterAgent | 报告撰写、脚本生成、报告修改 | Fast | 包装 ReportingService + ScriptService |
| DirectorAgent | 协调各 Agent、任务分派 | — | Agent 注册表 + dispatch 模式 |

**产品价值：** 多 Agent 架构是当前 AI 产品的主流范式，面试时展示对 Agent 协作模式的理解是加分项。同时，分工后每个 Agent 的 prompt 更聚焦，输出质量更高。

**实现要点：**
- `agents/base.py` — `BaseAgent` 抽象基类（name, capabilities, execute）+ `AgentResult` 统一结果包装
- `agents/planner.py` — `PlannerAgent`，支持 plan/analyze/gain 三种 action
- `agents/researcher.py` — `ResearcherAgent`，支持 search/research 两种 action
- `agents/critic.py` — `CriticAgent`，封装报告质量评估
- `agents/writer.py` — `WriterAgent`，支持 report/revise/script 三种 action
- `agents/director.py` — `DirectorAgent`，Agent 注册表 + dispatch 分派模式
- `agent.py` — `DeepResearchAgent` 初始化时创建 Director 并注册所有 Agent
- 渐进式迁移：现有编排逻辑保留，Agent 层可逐步接管

### 优化方向 3.2：混合记忆管理 ✅ 已实现

**核心思路：** Agent 需要短期记忆（当前对话上下文）和长期记忆（提炼后的核心知识）。

| 记忆类型 | 内容 | 生命周期 |
|----------|------|----------|
| 短期记忆 | 当前研究轮次的搜索结果、中间状态 | 单次任务 |
| 长期记忆 | 提炼后的关键发现、已验证的事实、用户偏好 | 跨任务持久化 |
| 外部记忆 | 向量数据库中的历史报告、知识库 | 持久化 + 检索 |

**产品价值：** 记忆管理是 Agent 产品的核心基础设施。实现后可以支撑"连续对话式研究"（用户追问时 Agent 能记住之前的研究结果），这是竞品难以快速复制的能力。

**技术要点：**
- 短期记忆：当前的 `SummaryState` 已经部分实现，可以扩展为更结构化的上下文窗口
- 长期记忆：每轮研究结束后，提取关键实体和结论，存入结构化存储
- 外部记忆：历史报告向量化，支持 RAG 检索

**实现要点：**
- `services/memory_manager.py` 新增 `MemoryManager` 类，管理记忆的提取、持久化、检索
- 使用 LLM 从研究总结中提取结构化记忆（key_findings、entities、keywords）
- 基于关键词匹配的简易检索机制（Jaccard 相似度），按相关度排序
- 记忆以 JSON 文件持久化，索引文件管理所有记忆条目
- `agent.py` 在规划前检索相关记忆注入 prompt，研究后保存长期记忆
- 配置项：`enable_memory`（默认 True）

### 优化方向 3.3：智能终止策略 ✅ 已实现

**核心思路：** 在 LLM 判断基础上增加定量信息增益指标，双重保障搜索终止决策。

**产品价值：** 避免"搜不够"（质量差）和"搜过头"（浪费成本）两个极端，实现成本和质量的最优平衡。

**技术要点：**
- 每轮搜索后计算"新增信息量"（新实体数、新观点数）
- 当连续 N 轮新增信息量低于阈值时，判定为"信息饱和"
- 结合 token 预算和时间预算做综合决策

**实现要点：**
- `services/planner.py` 新增 `calculate_information_gain()` 方法，基于关键词重叠度计算信息重复度
- `agent.py` 在精炼循环中，每轮新任务完成后检查信息增益
- 如果信息重复度超过阈值（默认 80%），即使 LLM 建议继续也终止
- 配置项：`min_information_gain`（默认 0.8）
- 双重终止机制：LLM 定性判断 + 定量信息增益指标

## 四、Prompt 工程：从"通用指令"到"精细化设计"

### 优化方向 4.1：搜索决策 Prompt

**参考 DeepSearch 的结构化输出设计，为搜索决策 Agent 设计专用 Prompt：**

- 明确角色身份（行业研究员）
- 输入已收集信息摘要 + 历史搜索记录
- 输出结构化决策：是否继续搜索、继续原因、下一轮搜索关键词
- 要求搜索关键词多样化（覆盖行业规模、竞争格局、技术动态、政策环境等）
- 禁止重复历史搜索记录

### 优化方向 4.2：结构化输出策略选择

**当前项目使用 XGrammar 约束解码（100% 结构正确 JSON），这是一个技术亮点。** 但 DeepSearch 文章提到了自定义格式 `[field] value` 的优势：

- 对特殊字符、换行符更鲁棒
- 调试时人眼可读性更好
- 不依赖模型的 JSON 语法能力

**建议：** 保持当前 XGrammar 方案（这是技术亮点），但在搜索结果筛选等非关键路径上，可以考虑自定义格式作为降级方案。

## 五、用户体验：从"工具"到"产品"

### 优化方向 5.1：研究过程透明化 ✅ 已实现

**核心思路：** 让用户看到 Agent 的"思考过程"，而不只是一个等待动画。

**产品价值：** 透明度建立信任。用户看到 Agent 在"认真研究"，会更有耐心等待，也更信任最终结果。

**技术要点：**
- 当前已有 SSE 事件流（stage_change、task_status、tool_call），基础良好
- 可以进一步展示：每轮搜索的关键词、发现了什么新信息、为什么决定继续/停止搜索
- 前端 TerminalLog 组件已经具备展示能力

**实现要点：**
- `agent.py` 新增细粒度 SSE 事件：`search_query`（搜索关键词）、`sources`（含 result_count）、`task_findings`（任务关键发现）
- 迭代精炼阶段发射 `refine_round`、`refine_saturation` 展示深度搜索进度
- 报告精炼阶段发射 `report_refine` 展示 Critic 评分和改进建议
- `App.vue` 处理所有新事件，在 TerminalLog 中实时展示

### 优化方向 5.2：用户可控性增强

**核心思路：** 在自动化流程中引入用户决策点。

**产品价值：** "全自动"和"可调控"之间，用户更信任后者。关键决策点让用户参与，既提升体验又降低风险。

**可能的决策点：**
- 搜索策略确认：Agent 生成搜索计划后，用户可以修改关键词再执行
- 报告大纲确认：生成报告前，用户可以调整大纲结构
- 脚本风格选择：单人/双人播客、正式/轻松风格

## 七、面试话术建议

**当被问到"这个项目的技术亮点是什么"时：**

> 核心亮点有七个。第一是 **Plan-and-Solve 工作流模式**——先规划再执行，确保研究的系统性和可追溯性。第二是 **XGrammar 约束解码**——通过正则文法约束 LLM 输出，实现 100% 结构正确的 JSON。第三是 **Smart/Fast 双层模型架构**——规划和报告用强推理模型，摘要和脚本生成用快速模型，在质量和成本之间取得平衡。第四是 **迭代式深度搜索**——LLM 自动分析信息缺口、生成补充搜索任务，直到信息饱和。第五是 **报告 Self-Refine**——Critic Agent 评估报告质量并给出改进建议，Writer Agent 据此修改，形成质量自闭环。第六是 **多智能体架构**——将服务层抽象为正式的 Agent 接口（Planner/Researcher/Critic/Writer），由 Director 统一协调，提升可扩展性。第七是 **智能终止策略**——在 LLM 定性判断基础上增加定量信息增益指标，双重保障搜索终止决策的合理性。

**当被问到"如果继续优化，你会怎么做"时：**

> P0-P3 的九个优化方向已经全部落地。接下来可以从两个方向深化：**产品化**——引入用户可控性（搜索策略确认、报告大纲确认、脚本风格选择），让"全自动"升级为"可调控"。**性能优化**——引入并发限流（当前是每任务一线程，可以改为线程池）、搜索结果缓存、TTS 流式合成等，提升端到端响应速度。
