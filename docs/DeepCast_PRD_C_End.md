# DeepCast 产品需求文档 (PRD)

## 1. 产品概述

### 1.1 产品定位
**DeepCast** 是一款面向 C 端高价值人群的**“知识焦虑破局工具”**。它将长篇万字研报、深度的前沿概念、或碎片化的硬核知识，自动化提炼并重绘为 **15分钟高质量双人对谈式播客 (Podcast)**。
核心理念：**释放“眼睛”与“整块时间”，用“耳朵”在碎片化场景中完成深度认知。**

### 1.2 目标用户与使用场景
*   **金融投资人 / 行业分析师**：
    *   *痛点*：每天面对海量行研报告，无法在有限时间内全部阅读。
    *   *场景*：在开车通勤、健身跑步时，听 DeepCast 深入剖析“固态电池最新产业链现状”的双人深度对谈。
*   **科研工作者 / 研究员**：
    *   *痛点*：需要跨界了解其他学科的最新进展，但看论文门槛极高且枯燥。
    *   *场景*：洗碗或做家务时，听 AI 对话拆解“AlphaFold 3 的核心算法与生物学意义”。
*   **一二线城市白领 / 终身学习者**：
    *   *痛点*：有极强的知识焦虑，买了很多专栏和电子书但大多“吃灰”，缺乏完整的阅读时间。
    *   *场景*：在地铁通勤的 30 分钟里，听一期关于“量子计算商业化落地”的生动播客，完成知识充电。

### 1.3 核心竞争力 (相比于纯文本总结与单口TTS)
*   **降维易读性**：双人对谈（Host & Guest）天然具备捧哏/逗哏逻辑，通过提问与解答，将生硬的知识转化为人类大脑更易接受的故事流。
*   **信息密度与深度**：非简单摘要，而是基于全网深度混合搜索 (Tavily/SerpApi) 与逻辑推理大模型，确保言之有物，拒绝AI常识废话。
*   **全自动工作流**：从“抛出概念”到“直接收听”，用户零干预，极大地降低了认知门槛。

## 3. 功能需求详细说明

### 3.1 核心主干流程 (Happy Path)
1.  **输入需求**：用户在 Web 端输入想要了解的复杂硬核主题（如：“解析2026年类脑计算的突破与瓶颈”）。
2.  **生成阶段 (黑盒/半透盒)**：前端展示进度条和极客风的 Terminal 实时日志，满足用户的“等待确定感”。
3.  **播放体验**：音频流式生成，第一段音频就绪后即可点击“播放”，黑胶唱片 UI 旋转，配套滚动阅读 Markdown 深度研报。
4.  **消费与分享**：听完后可下载 MP3 到本地，或分享生成页面给好友。

### 3.2 功能拆解清单

| 模块 | 功能名称 | 优先级 | 功能描述说明 |
| :--- | :--- | :--- | :--- |
| **输入端** | 核心主题探索 | P0 | 提供单行文本框输入主题，支持一键发送。 |
| **输入端** | 提示词市场/灵感库 | P1 | 针对不知如何提问的用户，提供“今日深度投资话题”、“科技前沿”等预设 Tag 快速点击补全。 |
| **工作流** | 深度思考可视化 | P0 | 在等待生成的数分钟内，用 Terminal UI 实时反馈 Agent 正在执行的动作（如：正在请求 Tavily... 正在生成大纲...），缓解等待焦虑。 |
| **音频端** | 黑胶沉浸播放器 | P0 | 提供支持播放/暂停、进度拖拽、倍速播放 (1.2x, 1.5x) 的音频播放组件。 |
| **音频端** | 流式分段播报 | P1 | 突破 TTS 慢速瓶颈，按对话句柄并行分块合成，实现首段音频极速出声（TTFA: Time to First Audio）。 |
| **阅读端** | 同步研报沉淀 | P1 | 播放器旁边提供最终的 Markdown 深度研究报告，供需要“深究”细节的用户图文对照阅读。 |
| **账号与存储**| 播客历史架 | P2 | 记录用户历史生成的所有播客项目，支持云端随时回听。 |

## 4. 商业化方案设计 (MaaS & SaaS)

采用 **Freemium (免费增值)** 商业模型，兼顾用户拉新与高净值用户变现。

*   **Free (基础版) - 免费引流**
    *   每月赠送 3 期免费生成额度（每期限制 5 分钟短播客）。
    *   限制浅层网络搜索，默认音色。
*   **Pro (深度版) - $9.9 / 月** (针对知识焦虑用户、通勤族)
    *   每月 30 期深度生成额度（每期 15-20 分钟），支持极速音频队列。
    *   深度混合搜索 Agent（支持10倍并发节点），生成更详尽的行研级播客。
    *   支持选择多种音色组合（如：严肃新闻腔、幽默脱口秀风）。
*   **Max/Enterprise (专业版) - 按量计费或 $39 / 月** (针对金融/研究机构)
    *   支持**文档上传解析 (PDF/Word/网页链接)** 转播客。
    *   开放 API 接口（MaaS层输出），支持与其他自动化工作流集成。

## 5. 数据埋点与核心指标 (KPIs)

为了验证产品是否达到“AIGC产品经理”所要求的留存与体验指标，需搭建以下数据体系：

### 5.1 北极星指标 (North Star Metric)
*   **WAST (Weekly Average Session Time)**：活跃用户周均收听 AI 播客的总时长。此指标最能反映“填补用户碎片化整块时间”的核心价值。

### 5.2 体验与转化指标
*   **转化漏斗**： `输入主题 (100%)` -> `等待超过30秒 (90%)` -> `点击播放 (80%)` -> `完播率>80% (35%)`。
*   **TTFA (Time to First Audio)**：从用户提交主题，到第一段对话音频成功下发并可播放的平均耗时，目标控制在 30 秒内。
*   **生成失败率**：统计大模型超时、TTS 合成失败或脚本解析错误的报错比例。

### 5.3 商业化指标
*   **免费转付费转化率 (Free-to-Paid Conversion)**：试用完免费额度后，转化为 Pro 用户的比例。
*   **复购率 (Retention)**：连续两个月续费 Pro 会员的用户比例。

### 5.4 数据驱动的产品决策案例

| 决策点 | 数据/反馈来源 | 决策内容 | 量化结果 |
|-------|-------------|---------|---------|
| **进度条初始值** | 用户测试反馈"以为卡住了" | 从 0% 起步改为 2% 起步 | 消除静止错觉，等待留存率提升 |
| **Terminal UI** | 用户访谈"不知道 AI 在做什么" | 实时展示 Agent 内部动作 | 将"黑盒等待"变为"半透明过程" |
| **双输出形态** | 用户场景分析（通勤 vs 桌面） | 同时提供音频 + Markdown 报告 | 满足不同消费场景，提升内容利用率 |
| **取消功能** | 等待阶段流失率分析 | 添加取消按钮 + 确认弹窗 | 给用户控制感，减少强制关闭 |
| **黑胶唱片 UI** | 播放行为分析"只是在播放文件" | 设计旋转唱片 + 品牌标识 | 将"播放文件"变为"收听播客"的仪式感 |

## 6. AI 模型能力与 Prompt 设计

### 6.1 模型选型策略：双层模型架构

DeepCast 采用 **Smart / Fast 双层模型** 策略，根据任务复杂度匹配不同模型，兼顾质量与效率：

| 层级 | 模型 | 定位 | 应用场景 | 选型理由 |
|------|------|------|---------|---------|
| **Smart** | `ecnu-reasoner` | 深度推理 | 任务规划、报告生成 | 复杂逻辑拆解需要强推理能力，规划质量直接决定研究深度 |
| **Fast** | `ecnu-max` | 快速响应 | 搜索摘要、脚本生成 | 高并发场景下需要低延迟，摘要任务对推理深度要求较低 |

**关键决策：** 规划阶段使用 Smart 模型是因为"任务拆解"的质量直接决定了后续所有环节的上限——错误的研究方向无法通过后续摘要来弥补。而逐任务摘要使用 Fast 模型是因为每个子任务的搜索结果已经提供了充分上下文，摘要质量更多取决于信息提取能力而非推理深度。

### 6.2 Prompt Engineering 方法论

#### 角色人格设计

DeepCast 的双主持人（Host 苏打 + Guest 冰糖）并非简单的"男声+女声"组合，而是经过精心设计的**认知分工**：

| 维度 | Host 苏打 | Guest 冰糖 |
|------|----------|-----------|
| **认知角色** | 代表普通听众视角 | 代表领域专家视角 |
| **提问模式** | "这个是什么？""为什么重要？" | "核心原理是……""关键在于……" |
| **情感基调** | 好奇、幽默、偶尔惊叹 | 沉稳、专业、偶尔幽默 |
| **年龄/性别** | 二十多岁男性 | 三十岁左右女性 |

**设计原则：** Host 的提问必须是听众会问的问题（降低认知门槛），Guest 的回答必须有专业深度（保证信息密度）。这种"好奇者-解答者"的对话结构天然适配知识传递场景。

#### 情感标注体系

脚本生成时，每轮对话必须标注 `emotion`（情绪状态）和可选的 `audio_tag`（音频标签），而非仅生成纯文本：

```json
{
  "role": "Host",
  "content": "这也太厉害了吧！",
  "emotion": "忍不住惊呼，语速加快，音量提高",
  "audio_tag": "提高音量"
}
```

**为什么需要情感标注：** 传统 TTS 将每句话视为独立输入，无法感知上下文的情感变化。通过在脚本层面标注情绪，TTS 模型可以基于 `emotion` 字段生成导演模式的 style 指令，实现"同一角色在不同语境下有不同的情绪表达"。

#### 结构化输出保障

脚本生成使用 XGrammar 约束解码（`response_format: json_schema`），保证 LLM 输出 100% 符合预定义的 JSON Schema。这是**工程层面的可靠性保障**——不依赖 LLM 的"自觉"来输出正确格式，而是通过 token 级别的约束强制输出结构正确。

### 6.3 TTS 模型能力边界

MiMo-V2.5-TTS 提供三级控制粒度：

| 级别 | 控制方式 | 粒度 | DeepCast 的使用 |
|------|---------|------|----------------|
| **L1 预置音色** | `audio.voice` 参数 | 角色级 | 降级方案：使用预置的"苏打/冰糖"音色 |
| **L2 文本音色设计** | 自然语言描述音色 | 角色级定制 | 主方案：通过 VoiceDesign 模型自定义音色 |
| **L3 导演模式** | 角色/场景/指导三维度 | 句子级情感 | 每句台词动态生成 style 指令 |

**能力边界认知：**
- TTS 的情感控制依赖于 `user` 消息中的自然语言指令，而非 `assistant` 消息中的文本内容
- 音频标签（`(轻笑)`、`(叹气)`）需要嵌入 `assistant` 消息文本中，与自然语言控制放在不同位置
- VoiceDesign 模型不支持预置音色和音色复刻，三种模型的能力互斥

## 7. 后续迭代规划 (Roadmap)
*   **v1.0 (当前)**：主题直出双人播客，MVP 跑通搜索到合成的心智。
*   **v1.5 (拓展输入)**：支持用户丢入公众号文章链接、PDF 财报，针对特定信源生成播客。
*   **v2.0 (人机交互)**：生成报告大纲后增加 Halt 环节，允许用户“审批”或修改大纲，Agent 再根据修改后的指令生成剧本与音频。
*   **v2.5 (音色广场)**：引入声音克隆技术，允许用户导入特定人的声音作为 Host，打造个人的专属“AI 知识伴游”。
