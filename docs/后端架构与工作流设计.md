# DeepCast 后端架构与工作流设计文档

本文档全面梳理了 DeepCast 后端核心架构的工作流机制、近期的代码演进汇总，并重点详细记录了“流式任务取消机制”所经历的多次推翻与迭代，以及对导致无法取消的“根本原因”的剖面分析。

## 0. Agent 设计模式与技术决策

### 0.1 Plan-and-Solve 多智能体工作流

DeepCast 采用 **Plan-and-Solve** 模式将复杂任务拆解为可并行执行的子任务，这是区别于"单 LLM 一次性生成"的核心架构决策：

```
用户输入主题
  → Memory Manager → 检索相关历史研究记忆
  → Planning Agent（Smart LLM）→ 3-5 个 TodoItem 子任务
  → [并行] Search Agent × N（含 LLM 结果过滤）+ Summarization Agent × N
  → Refine Agent（Smart LLM）→ 分析信息缺口 → 补充搜索任务（迭代至饱和）
  → Reporting Agent（Smart LLM）→ Self-Refine：初稿 → Critic 评估 → 修改
  → Memory Manager → 提取关键发现持久化
  → Script Agent（Fast LLM）→ 双人对话 JSON 脚本（含 emotion + audio_tag）
  → Audio Agent（MiMo TTS 导演模式）→ 逐句语音合成
  → Synthesis Agent（FFmpeg）→ 最终播客 MP3
```

**为什么不用单 Agent 一次性生成：**
- 复杂主题（如"量子计算商业化"）需要多角度搜索，单 Agent 无法并行
- 研究深度取决于搜索覆盖面，并行 3-5 个子任务比串行搜索效率提升 3-5 倍
- 每个阶段的输出质量需要独立验证（规划质量 → 搜索覆盖度 → 摘要准确性 → 报告连贯性）

**为什么选择 Plan-and-Solve 而非 ReAct：**
- ReAct（Reason + Act）模式适合需要动态调整策略的场景（如多轮对话）
- DeepCast 的任务是线性的：规划 → 搜索 → 摘要 → 报告 → 脚本 → 音频
- Plan-and-Solve 的优势在于**规划阶段一次性确定所有子任务**，后续阶段可以完全并行执行

### 0.2 结构化输出：XGrammar 约束解码

DeepCast 的关键输出（任务规划、对话脚本）使用 XGrammar 约束解码保证 100% 结构正确：

**技术原理：** XGrammar 在 token 生成阶段通过有限状态机约束每一步的 token 选择，确保最终输出严格符合预定义的 JSON Schema。这与传统的"生成后解析"方式有本质区别——传统方式依赖 LLM 的"自觉"来输出正确格式，而 XGrammar 在生成层面强制保证格式正确。

**应用场景：**

| 场景 | Schema | 约束内容 |
|------|--------|---------|
| 任务规划 | `PLANNER_JSON_SCHEMA` | 3-5 个任务，每个必须有 title/intent/query |
| 节目蓝图 | `SCRIPT_BLUEPRINT_JSON_SCHEMA` | 必须包含 Hook、3 个主体段、转场、收尾和 CTA |
| 对话脚本 | `SCRIPT_JSON_SCHEMA` | role 必须是 Host/Guest，emotion 必填，audio_tag 可选 |

**决策权衡：**
- **优势：** 消除 JSON 解析错误（传统方式约 5-10% 的输出格式不正确），减少重试次数，降低延迟
- **劣势：** XGrammar 增加了 token 生成的计算开销（约 10-15%），且需要精确定义 Schema
- **结论：** 对于 DeepCast 的场景（格式正确性 > 生成速度），XGrammar 是正确的选择

### 0.3 模型分层策略

| 任务类型 | 模型 | 理由 |
|---------|------|------|
| 任务规划 | Smart (ecnu-reasoner) | 规划质量决定研究上限，需要深度推理 |
| 报告生成 | Smart (ecnu-reasoner) | 需要综合多个摘要，形成连贯论述 |
| 逐任务摘要 | Fast (ecnu-max) | 搜索结果已提供充分上下文，摘要更依赖信息提取而非推理 |
| 脚本生成 | Fast (ecnu-max) | 高并发场景（10-20 句对话），低延迟是关键 |

**关键洞察：** Smart 模型用于"决定做什么"（规划 + 报告），Fast 模型用于"执行具体任务"（摘要 + 脚本）。这种分层策略在质量与效率之间取得了平衡。

### 0.4 多智能体抽象层

DeepCast 在服务层之上引入了正式的 Agent 抽象层（`agents/` 目录），将各服务封装为具有统一接口的专业 Agent：

```
DirectorAgent（协调器）
  ├── PlannerAgent   → 包装 PlanningService（plan / analyze / gain）
  ├── ResearcherAgent → 包装 SearchService + SummarizationService（search / research）
  ├── CriticAgent    → 包装 ReportingService 的评估逻辑（critique）
  └── WriterAgent    → 包装 ReportingService + ScriptGenerationService（report / revise / script）
```

**设计原则：**
- `BaseAgent` 抽象基类定义 `name`、`capabilities`、`execute(context) -> AgentResult` 统一接口
- `AgentResult` 统一包装执行结果（success / data / events / metrics）
- `DirectorAgent` 通过注册表管理所有 Agent，提供 `dispatch(agent_name, context)` 分派方法
- 渐进式迁移：现有 `DeepResearchAgent` 编排逻辑保留，Agent 层可逐步接管

## 1. 后端改动汇总 (Change Log)

在近期的重构与优化中，后端主要修复了从底层环境到应用架构的多项问题，改动包含以下核心方向：

1. **框架机制更新**：
   - 使用 FastAPI 的 `@asynccontextmanager` `lifespan` 替换已弃用的 `@app.on_event("startup")`，确保应用启停生命周期的安全与规范。
   - `CORS Middleware` 的来源配置调整为从 `config.py` 中的 `cors_origins` 动态读取，解除了原硬编码导致部分前端端口（如 `5174`）跨域拒绝的问题。
2. **多线程安全强化**：
   - 针对 Search 工具在并发规划时的竞态条件死锁风险，在 `services/search.py` 中引入 `threading.Lock()` 保驾护航。
3. **主生成器解耦**：
   - 重构了长达 350+ 行的 `agent.py::run_stream()` 庞大方法。将其按职责链硬拆分为四个阶段方法：`_stream_research_phase`, `_stream_report_phase`, `_stream_script_phase`, `_stream_audio_phase`，显著降低了圈复杂度。
4. **模型层修复与补调**：
   - 删除了 `__init__.py` 中不存在的 `SummaryStateInput`，修复了导入报错。
   - 弃用容易引发异常且复杂化的 `Annotated[list, operator.add]` 类型绑定，退回轻量可靠的内置原生 list 管理对象。
   - 补齐了 `env.example` 中的 `HOST`、`PORT`、`LLM_TIMEOUT` 等系统配置映射，后端能真正遵循系统环境。

## 2. 核心工作流设计 (Stream Workflow)

后端的中心思想是：**长耗时业务在后台同步线程池（ThreadPoolExecutor）执行，通过队列（Queue）跨线程推送到主事件循环（Event Loop），以此桥接 SSE（Server-Sent Events）实现平滑逐字流式输出**。

主要工作流分为六个阶段（Phases）：

### Phase 1: 规划与并行研究 (`_stream_research_phase`)
通过 `PlanningService` 对用户 Topic 进行拆解，生成 `TodoItem` 列表。每个子任务挂载到独立的挂载线程并发执行（`_execute_task`），同时调用搜索与总结模型。每个任务执行时：
- 使用 LLM 评估搜索结果质量，过滤低价值内容（`filter_search_results`）
- 发射 `search_query`、`sources`（含 result_count）、`task_findings` 等细粒度 SSE 事件
- 如果启用长期记忆，研究前检索相关历史记忆注入规划 prompt

### Phase 1.5: 迭代式深度搜索 (`_stream_refine_phase`)
初始任务完成后，进入迭代精炼阶段。LLM 分析已有研究总结，识别知识缺口，动态生成补充搜索任务：
- 使用 `PlanningService.analyze_and_refine()` 评估信息完整性
- 如果发现显著缺口 → 生成 2-3 个补充任务 → 并行执行搜索+总结
- 如果信息饱和 → 终止精炼，进入报告阶段
- 最大迭代轮次由 `max_research_refine_rounds` 配置控制（默认 2）
- 发射 `refine_round`、`refine_saturation` 等 SSE 事件

### Phase 2: 深度报告撰写 (`_stream_report_phase`)
归集所有研究总结（含精炼阶段的补充结果），调用 `ReportingService` 生成结构化 Markdown 报告。报告生成采用 **Self-Refine** 机制：
- 初稿生成后，Critic Agent 评估逻辑严谨性、数据支撑度、专业性等维度
- 如果评分 < 7 或 verdict 为 needs_revision，Writer Agent 根据反馈修改
- 迭代轮次由 `max_report_refine_rounds` 配置控制（默认 1）
- 发射 `report_refine` 事件展示精炼进度
- 报告完成后，如果启用长期记忆，提取关键发现持久化到记忆系统

### Phase 3: 播客脚本生成 (`_stream_script_phase`)
将结构化报告通过 `ScriptGenerationService` 的结构化输出机制，转写为双主播（Host 苏打 & Guest 茉莉）的对话剧本。脚本包含 `emotion`（情绪+语速节奏描述）和 `audio_tag`（细粒度音频标签）字段，用于驱动高质量语音合成。

### Phase 4: 音频生成与合成 (`_stream_audio_phase`)
拉起 `AudioGenerationService` 进行 TTS API 调用。语音合成采用**导演模式**：
- 为每个角色构建三段式 style 指令（角色/场景/指导），包含差异化语速和能量指导
- Host（苏打）：预置音色，语速偏快、节奏活泼
- Guest（茉莉）：预置音色，语速适中偏快、节奏明快
- VoiceDesign 模式下使用文本描述生成音色，同样包含语速指导
- 支持 emotion 和 audio_tag 驱动的动态风格控制
- 最后利用 FFmpeg 完成最终长音频的拼接

## 3. 取消机制的演进史与根本原因

DeepCast 开发周期内面临的最棘手问题是 **“点击取消后，后端的报告生成和 TTS 合成依然在跑”**。对此，后端的处理逻辑经过了数次迭代和底层重修。

### 尝试一：密集设签法 (is_cancelled)
- **初始思路**：在所有的 yield 前、Phase 接口间、以及部分长循环中补充 `if self.is_cancelled(): return` 或 `break`。
- **失败原因**：在执行 `generate_report`、`generate_script` 等阶段时，系统正处于 HTTP 连接（OpenAI 客户端）的同步 **强阻塞** 中（通常耗时 10-30 秒不返回）。主背景线程因阻塞而暂停在 `next()` 调用，导致所有新加入的 `is_cancelled()` 检查在请求返回之前，根本**没有机会得到执行**。

### 尝试二：守护线程外壳脱壳 (`_cancellable_call` 封装)
- **改进思路**：由于长耗时代码段是同步阻塞的，于是引入了 `_cancellable_call` 辅助方法。利用再起一个背景守护线程跑 LLM Api，主控制器按 `done.wait(timeout=0.5)` 进行轮询监测。（见前两版代码的 `CancelledException` 实现）。
- **连带动作**：尝试加入 HTTP client 强制 shutdown 来拦截网络。
- **失败原因**：方案极度复杂，且容易造成未知连接泄露或错误吞噬。此时我们发现即使用了极其激进的挂断控制，取消**还是不一定能完全触发**。

### 🔬 真正的根本原因剖析 (Root Cause)

问题的核心根本不在于 `is_cancelled()` 轮询得多还是少，而在于一场 **关于生命周期管理的极速竞态条件问题（Race Condition）**。

1. **断联瞬间**：前端一旦呼叫取消，往往第一动作是调用 `abortController.abort()` 强行打断 SSE 数据流连接。
2. **FastAPI 的清理回落**：底层 SSE 断开触发 `main.py` 的 `yield` 循环直接抛出并走入 `finally` 清理块。
3. **致命的顺序错误**：
   在最初的版本中，`finally` 块的执行顺序为：
   ```python
   # 错误逻辑：
   monitor_task.cancel()  # 1. 关掉断联捕捉器
   executor.shutdown(wait=False)
   _active_agent["current"] = None # 2. 将全局上下文立刻清空！
   ```
4. **后置求救 API 被丢弃**：由于前端同时发起了一次 `POST /research/cancel` 的明确止损请求，但此 HTTP POST 请求到达后端时（往往存在 100ms 量级的延时），刚才的 `finally` **早就把 `_active_agent["current"]` 给置为了 `None`**。
5. **结果**：POST API 发现没获取到上下文，回传了 `no_task` 提示，导致最重要的真正取消命令 `agent.cancel()` 【**从未被执行过**】。全盘的 `_cancel_event` 始终为 False。即便线程切回去执行，系统依旧以为是绿灯放行。

### 🎯 最终极简解决方案 (Minimal Elegant Fix)

查明真因后，移除了过度设计的 `_cancellable_call`、HTTP 断联机制以及复杂的 `CancelledException`，回归最底层的机制本源。

**终局修正：**
在 `main.py` 的 `event_iterator` 清理块的第一行，强制绑定 `agent.cancel()`：
```python
finally:
    # 确保取消信号被设置 —— 这是取消机制的核心防线！
    # 前端 abort SSE 后，哪怕 monitor 还没来得及捕捉，
    # 哪怕前端的 explicit /cancel API 还没抵达。从连接层面断定必须停止当前任务：
    agent.cancel()  # 🌟 关键的一步
    
    monitor_task.cancel()
    # ...后续安全清理动作...
    _active_agent["current"] = None
```

凭借这行代码，系统确保了：**无论 SSE 异常断联还是明确取消，事件钩子都能必然触达 Agent 的信道**。
配合在各大阻塞点外围（Phase 入口/出口等）最质朴的 `is_cancelled()` 协程校验，达到了**极低的维护成本和 100% 成功率的系统叫停**。守护线程的 LLM API 强阻塞会在最后一次完成时，静默地发现全局取消并放弃结果抛弃写入，流程完美闭环。
